** What This Produces

This contract validates a recoverable plan ledger. It defines the theory of a
plan record: a bounded objective, a scope, optional specification references,
recoverable state, evidence, and a next action. Runtime commands and
client-specific workflows are out of scope for this contract.

The intended output is a single recoverable task heading with:

- a native Org workflow state and progress cookies;
- identity, objective, scope, status, recovery properties, and only relevant specification references for later agents;
- a lifecycle checklist that shows where the plan stands;
- Context, Checkpoints, Validation, Evidence, Evidence Loop, Reflection, and Recovery sections;
- concrete evidence, reflection answers, and recovery anchors that can be inspected without a live LLM.

** Definition

A Plan is a durable ledger of work state. It is written after or during work so
another agent can recover the objective, current status, relevant specification
references, evidence, and next action without relying on conversation history.

A Plan is not the governing product, behavior, decision, architecture, test, or
business-development specification. Those records live under purpose-specific
contracts such as =agent.prd.v1=, =agent.bdd.v1=, =agent.adr.v1=, =agent.sdd.v1=,
=agent.tdd.v1=, or =agent.bdr.v1= and are linked only when relevant.

A Plan may link to governing specifications when they are relevant. Agents decide whether SDD, BDD, TDD, or BDR applies before writing the plan, but unused specification families must be omitted instead of recorded as placeholder =none= rows or properties.

Use this contract-owned choice window before creating or updating a Plan. The choice is transient: select one specification or ask for details, then render only the plan content and references that are materially needed by the task.
