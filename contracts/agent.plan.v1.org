* agent-plan-v1
:PROPERTIES:
:CONTRACT_ID: agent.plan.v1
:TITLE: Agent Plan Ledger v1
:END:

** What This Produces

This contract validates a recoverable plan ledger. It defines the theory of a
plan record: a bounded objective, a scope, optional specification references,
recoverable state, evidence, and a next action. Runtime commands and
client-specific workflows are out of scope for this contract; storage placement
and filename shape are part of the contract because agents can create several
plan ledgers during one session.

The intended output is a single recoverable task heading with:

- a native Org workflow state and progress cookies;
- identity, objective, scope, status, recovery properties, and only relevant specification references for later agents;
- a lifecycle checklist that shows where the plan stands;
- Context, Checkpoints, Validation, Evidence, Evidence Loop, Reflection, and Recovery sections;
- concrete evidence, reflection answers, and recovery anchors that can be inspected without a live LLM.

** Definition

A Plan is a durable ledger of work state. It is written after or during work so
another agent can recover the objective, current status, relevant specification
references, evidence, and next action without relying on conversation history.

A Plan is not the governing product, behavior, decision, architecture, test, or
business-development specification. Those records live under purpose-specific
contracts such as =agent.prd.v1=, =agent.bdd.v1=, =agent.adr.v1=, =agent.sdd.v1=,
=agent.tdd.v1=, or =agent.bdr.v1= and are linked only when relevant.

A Plan may link to governing specifications when they are relevant. Agents decide whether SDD, BDD, TDD, or BDR applies before writing the plan, but unused specification families must be omitted instead of recorded as placeholder =none= rows or properties.
** Specification Choice Reasoning
Use this contract-owned choice window before creating or updating a Plan. The
choice is transient: select one specification or ask for details, then render
only the plan content and references that are materially needed by the task.

#+BEGIN_SRC org-contract :type agent-interactive
id: specification
method: choice
stage: pre-capture
group: SPEC
create: deferred
target: agent.plan.v1
info: Select one specification if you already know the terms. If not, choose ? to expand details before selecting.
categories: 1=SDD,2=BDD,3=TDD,4=BDR,5=TASK,?=detail
details:
|n|id|contract|full|use-if|
|1|SDD|agent.sdd.v1|Specification-Driven Development|scope/design/API/schema boundary changes|
|2|BDD|agent.bdd.v1|Behavior-Driven Development|behavior/scenario/acceptance unclear|
|3|TDD|agent.tdd.v1|Test-Driven Development|failing test/regression gate drives work|
|4|BDR|agent.bdr.v1|Business Development Representative|market/customer/revenue/outreach hypothesis|
|5|TASK||Task Plan|plain checklist/reflection is enough|
#+END_SRC
** Filename And Placement

Plan files must live under =artifacts/org/flow/plans/= or a project-local
equivalent ending in =org/flow/plans/=. A plan filename must match
=agent-plan-*.org= and should be derived from the task title that a future agent
can recall, for example =agent-plan-tighten-asp-org-recall-flow.org=. A client
session id may be appended as a disambiguating suffix, but it is not the recall
key. The capture command materializer enforces this placement for
=agent.plan.v1= when =--target-file= is provided. This keeps temporary task
placeholders such as =current-agent-task.org= out of the durable plan ledger
namespace.

The template root heading uses a placeholder title. Agents must pass
=--title "<task title>"= when materializing =agent.plan.v1= so the rendered plan
heading and =ID= property can be recovered by the =asp org recall plans= flow
after chat compaction. Rust owns Org discovery and contract filtering; Python
=asp-memory-engine= owns plan ranking, including memory/scoring math, but it
does not scan Org files.
