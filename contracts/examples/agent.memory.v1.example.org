** Claim

Org records structured work metadata and memory candidates, but it is not the
memory algorithm. Long-term memory systems consume validated workspace records
and dispatch receipts downstream.

** Source

- =docs/20-workspace/20.10-org-native-agent-workspace.org= states that Org
  records plans, evidence, trace records, and memory candidates.
- =docs/40-rfcs/40.10-org-native-agent-work-protocol.org= keeps memory dispatch
  as a downstream backend concern.
- =docs/20-workspace/20.40-org-memory-backend.org= defines
  =marlin-org-memory= as the in-memory =AgentWorkspace= backend.

** Recall

- Recall when designing memory candidate records, memory dispatch receipts, or
  agent-facing recall prompts.
- Recall when a contract only stores =Source/Trust/Retention= and does not tell
  an agent what claim should be remembered or how to retrieve it.
- Use query terms from =RECALL_QUERY= before broad workspace search.
- Treat =RECALL_QUERY= as a hybrid recall frontier index. =fd=, =rg=, and Org
  query may use it to find candidates, but graph recall must rerank by typed
  project/session/content relationship facts and evidence.

** Evidence

- Parser-owned Org facts expose memory candidates as workspace records.
- Workspace patches can mark memory candidates and return memory dispatch
  receipts.
- The memory backend can query typed nodes, render compact views, apply typed
  patches in memory, and emit receipts.
- Evidence entries should be precise enough to support a compact context pack
  without reopening an entire session transcript or memory shard.

internal-doc; backed by project architecture and RFC documents.

** Retention

- Keep for the project lifecycle because it defines the memory boundary.
- Supersede only when a later RFC changes whether Org owns memory algorithms or
