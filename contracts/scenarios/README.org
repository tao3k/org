* ASP Contract Scenarios

This directory contains replayable Org scenario contracts for ASP architecture
flows that are larger than one schema file but still need to be auditable by
agents and humans.

The files here are not generic prose notes. Each scenario should include:

- scenario metadata in Org properties
- an =org-contract= block that defines the interaction surface
- compact agent-facing render examples
- failure modes
- validation expectations
- stable links to related schemas, RFCs, or artifacts when available

** Directory Lint Resolution

Contract references should be resolved from the Org files themselves, not from
hand-maintained command-line registration. A =CONTRACT_ORG= value such as:

#+BEGIN_SRC org
:CONTRACT_ORG: [[../basic.template.v1.org][basic.template.v1]]
#+END_SRC

already contains the agreement. The command line should not repeat that
agreement as a separate registry argument.

The desired public lint surface is the contract-library directory:

#+BEGIN_SRC sh
asp org lint languages/org/contracts/
#+END_SRC

When the input path is a directory, the linter should:

- discover all =.org= files under =languages/org/contracts/=, including
  nested directories such as =scenarios/=;
- treat those files as one contract-library lint target set;
- parse each target file's =CONTRACT_ORG= links;
- resolve each referenced contract path relative to the referencing file;
- recursively load referenced contract files;
- index loaded contracts by their declared =CONTRACT_ID=;
- validate each target document against the contract graph declared by that
  document;
- report failures against the target file that declared or violated the
  contract.

In this model, any in-memory "registry" is only an implementation cache built by
scanning the input directory and following paths declared in the Org documents.
It is not a user-facing agreement and not something an agent should maintain.

#+BEGIN_SRC sh
asp org lint languages/org/contracts/
#+END_SRC

The =orgize= lint loader should build that contract graph from the directory
and the files' own relative =CONTRACT_ORG= links. If this command reports that a
declared contract cannot be found in a preloaded registry, that is a loader
regression rather than a scenario-file authoring requirement.

If scenario-specific contracts become reusable beyond these files, promote them
into first-class contract registry files under =languages/org/contracts/=.
