** What This Produces
This contract validates ASP Org-specific requirements for an Org skill subtree.
Pair it with =skill.v1= by listing both contracts as =CONTRACT_ORG= properties
on the same subtree. Shared skill quality belongs in =skill.v1=; this contract
adds only semantic-AST query boundaries, command surfaces, state paths, and
recovery behavior.

The intended output is an ASP Org skill with:
- state workflow steps driven by =asp org= query/trace output;
- validation checks and pressure-scenario guidance;
- adversarial review criteria for model-audited quality;
- provider-boundary rules that prevent SDD, Wendao, or client planning labels
  from becoming hidden query modes.

** ASP Org
:PROPERTIES:

:END:

*** Use Boundary
Use this skill when the task depends on durable Org planning state, Org
contracts, Org template outputs, TODO headlines, checklist items, property
drawers, planning timestamps, links, or source blocks that should be queried
through =asp org=.

Do not use this skill when the task is generic Markdown authoring, free-form
chat planning, or client-local todo management that does not need Org AST facts
or Org contract validation.

*** State Workflow
ASP materializes the Org resource tree when project runtime state is installed
or synchronized. Start by resolving the writable artifact root, then ask the
provider what Org facts exist and let the result shape the next action. The
provider exposes =asp org query= for semantic AST facts. The provider exposes
=asp org contract trace= for contract validation.

Use =asp org recall plans= before creating a new plan so the current agent session can resume related tasks, checklist state, and properties before branching new work.
#+begin_src shell
asp paths --get orgArtifacts
ORG_ARTIFACTS="$(asp paths --get orgArtifacts)"
asp org recall plans
asp org recall plans --checkpoint-sync
asp org guide
asp org query --kind task --field todo=TODO --workspace . --view metadata
asp org query --kind checklistItem --field checked=false --workspace . --view metadata
asp org query --kind property --field key=<KEY> --workspace . --view metadata
asp org query --kind property --field key=GOVERNING_CONTRACT --workspace . --view metadata
#+end_src

Archive lane:
- Resolve the writable artifact root with =asp paths --get orgArtifacts= before archiving from a non-root working directory.
- Before creating a new plan or task branch, recall candidate plans for the current agent session with =asp org recall plans=. Supported current-session envs are =CODEX_THREAD_ID=, =CLAUDE_CODE_SESSION_ID=, and =CLAUDE_CODE_REMOTE_SESSION_ID=; the active agent platform is expected to expose one non-empty value. There is no generic =AGENT_SESSION_ID= or =SESSION_ID= fallback. Use explicit =--session= only for cross-session lookup. Codex subagents have their own =CODEX_THREAD_ID=, so parent prompts must pass the root session explicitly or set =ASP_ROOT_SESSION_ID= before child recall. Keep =SESSION_ID= as the root/main task session in Org properties, and record branch edges with =ROOT_SESSION_ID=, =PARENT_SESSION_ID=, =CHILD_SESSION_ID=, or =SUBAGENT_SESSION_ID= when a plan spans child agent threads. Recall output is the reasoning tree entry point: text output starts with compact =|next=, =|why=, =|query=, =|evidence=, and =|candidate= rows. The =|why= row is the memory/graph selection receipt: session match, selected sources, task hits, checkpoint hits, alternatives, and memory transport. The executable resume surface is the single =|query= row, normally =asp org query <agent-plan-id> recovery evidence next-action=. JSON keeps full scores and per-plan =selectionReceipt= for tooling. Ordinary recall is read-only; add =--checkpoint-sync= only when you intend to persist a durable checkpoint into the SQLite-backed memory index.
- Query active Org facts first, such as =asp org query --kind task --field todo=DONE --workspace . --view metadata=, then inspect the returned selector before moving content.
- Preserve active layout under =archives= when possible.
- Add archive metadata such as =ARCHIVED_FROM= and =ARCHIVE_REASON= on the destination record instead of encoding archive status as a hidden search/query side effect.
- Do not archive by deleting AST facts discovered through =asp org query= without writing the destination archive record first.

Before creating or updating a Plan, ExecPlan, or governing specification family,
read the matching contract's =Definition= section. For =agent.plan.v1=, also
read =Specification Choice Reasoning= so the agent can decide whether SDD, BDD,
TDD, BDR, or none applies before writing the plan.

#+begin_src shell
asp org search owner .cache/agent-semantic-protocol/org/contracts/agent.plan.v1.org items --query "Specification Choice Reasoning SDD BDD TDD BDR" --workspace . --view seeds
#+end_src

*** Contract Capture
Use =asp org capture --contract CONTRACT_ID= for a non-mutating,
contract-checked Org entry plan. =CONTRACT_ID= must be a registry id such as
=agent.task.v1=, =agent.plan.v1=, =agent.sdd.v1=, =agent.adr.v1=,
=agent.bdd.v1=, =agent.tdd.v1=, =agent.bdr.v1=, =agent.prd.v1=, or
=agent.execplan.v1=. Capture kinds stay generic; contract semantics are carried
by the =CONTRACT_ORG= property injected by capture. The contract id must be
explicit; do not use positional aliases such as =task=, =plan=, or =sdd=.

#+begin_src shell
asp org capture --contract agent.task.v1 \
  --title <TITLE> \
  --target-file <ASP_ORG_ARTIFACTS_ROOT>/<TASK_SLUG>.org
#+end_src

*** Rules
- Preserve the semantic AST provider boundary: query Org tasks, checklist items, properties, planning timestamps, links, and source blocks as Org facts.
- Keep =ASP_ORG_SKILL.org= as the single state skill entry; do not reintroduce a wrapper skill.
- Do not edit the audited template output directly. Change this contract source and refresh the output through the Org provider path.
- Do not use =--kind agent-plan=; =agent-plan= is not a provider-owned capture kind.
- Do not add guide labels such as =agent-plan-*=, =wendao-task-*=, =wendao-orgid=, =sdd-property=, or =plan-record=.
- Do not encode SDD, Wendao, or task archive behavior as hidden search/query side effects. Express them as AST queries first, then route mutations through an edit-plan or AST-patch contract.

*** Search Route Choice Reasoning

Use this contract-owned choice window before the first ASP search/query command
for a new intent. The choice is transient and does not create a plan artifact.
Select the narrowest route justified by current evidence; if uncertain, inspect
the provider =routeGraph= or guide.

|3|KNOWN_SYMBOL|semantic-search-packet.v1#routeGraph|Known symbol|function/type/API symbol exists|

| Choice | Use if | First route | Avoid |
| KNOWN_SELECTOR | structuralSelector/nativeFactId exists | syntax-outline; query-code only if exact code is needed | prime, line-range selector |
| KNOWN_OWNER | owner path/file exists | owner-skeleton or owner-items | prime unless owner is stale |
| KNOWN_SYMBOL | function/type/API symbol exists | syntax query or owner-items | broad rg before syntax |
| KNOWN_DEPENDENCY | dependency/API package exists | dependency-topology/import-usage | workspace prime |
| FAILURE_FRONTIER | failure/test/policy diagnostic exists | failure-frontier/test-selection | raw read loop |
| BROAD_QUERY | terms but no stable anchor | search pipe | repeated prime/fzf |
| UNKNOWN_WORKSPACE | no reliable anchor | search prime/project topology | pretending anchors exist |
| NO_ASP | non-repo/prose task | no ASP | unnecessary search |

=recommendedNext=, =routeGraph=, =omit=, and =avoid= fields. Do not reconstruct
commands from =sourceLocatorHint= or =displayLineRange=. Do not create Org plan
artifacts for normal search routing.
