** What This Produces
This contract validates a software design description. An SDD records durable
architecture knowledge: system boundaries, stakeholders, requirements,
quality attributes, design concerns, architecture views, interfaces,
invariants, decisions, traceability, evidence anchors, review anchors, and
links that execution plans can use as governing context.

An SDD is not an implementation ledger. It does not track implementation
progress, command transcripts, or agent session state. Those records belong in
a Plan, ExecPlan, evidence-loop receipt, or adversarial-review receipt that
links back to the SDD.

Templates for this contract are projections of the required structure below.
Do not add SDD-only template sections unless this contract also defines them.
