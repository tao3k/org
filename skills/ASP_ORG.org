** Use Boundary
Use this skill when the task depends on durable Org planning state, Org contracts, Org templates, TODO headlines, checklist items, property drawers, planning timestamps, links, or source blocks that should be queried through =asp org=.

Do not use this skill when the task is generic Markdown authoring, free-form chat planning, or client-local todo management that does not need Org AST facts or Org contract validation.

** State Workflow
Start by materializing the ASP Org resource tree into the project state directory, then ask the provider what Org facts exist and let the result shape the next action. The provider exposes asp org query for semantic AST facts. The provider exposes asp org contract trace for contract validation.

#+begin_src shell
asp org capture init
asp org guide
asp org query --kind task --field todo=TODO --workspace . --view metadata
asp org query --kind checklistItem --field checked=false --workspace . --view metadata
asp org query --kind property --field key=<KEY> --workspace . --view metadata
asp org query --kind property --field key=GOVERNING_CONTRACT --workspace . --view metadata
#+end_src

The capture init command materializes Org resources into the ASP Org resource root. ASP resolves that root as =${PRJ_CACHE_HOME}/agent-semantic-protocol/org= when =PRJ_CACHE_HOME= is set, otherwise as =<git-toplevel>/.cache/agent-semantic-protocol/org=.

Agent-authored Org state belongs under the separate artifacts root returned by =asp paths --get orgArtifacts=. It resolves to =${PRJ_CACHE_HOME}/agent-semantic-protocol/artifacts/org= when =PRJ_CACHE_HOME= is set, otherwise to =<git-toplevel>/.cache/agent-semantic-protocol/artifacts/org=. Reference this materialized skill from AGENTS.md with =@.cache/agent-semantic-protocol/org/skills/ASP_ORG.org=.

Run asp paths --get orgArtifacts before writing agent-authored Org state from a non-root working directory.

State layout after =asp org capture init=:
- =.cache/agent-semantic-protocol/org= is the git-backed Org resource checkout.
- =.cache/agent-semantic-protocol/org/contracts= stores contract resources.
- =.cache/agent-semantic-protocol/org/templates= stores template resources.
- =.cache/agent-semantic-protocol/org/skills/ASP_ORG.org= is the single ASP Org skill entry.
- =.cache/agent-semantic-protocol/artifacts/org= stores agent-authored Org state.
- =.cache/agent-semantic-protocol/artifacts/org/flow/plans= stores plan records.
- =.cache/agent-semantic-protocol/artifacts/org/flow/sdd= stores SDD records.
- =.cache/agent-semantic-protocol/artifacts/org/flow/BDR= stores BDR records.
- =.cache/agent-semantic-protocol/artifacts/org/archives= stores completed, superseded, or inactive agent-authored Org state.

The archive root is =.cache/agent-semantic-protocol/artifacts/org/archives=, or the corresponding path under =$(asp paths --get orgArtifacts)/archives= when the artifacts root is resolved dynamically.

Archive records live under artifacts/org/archives.

Archive lane:
- Resolve the writable artifact root with =asp paths --get orgArtifacts= before archiving from a non-root working directory.
- Query active Org facts first, such as =asp org query --kind task --field todo=DONE --workspace . --view metadata=, then inspect the returned selector before moving content.
- Preserve active layout under =archives= when possible: move =flow/plans/<PLAN_SLUG>.org= to =archives/flow/plans/<PLAN_SLUG>.org=, SDD records to =archives/flow/sdd/=, and BDR records to =archives/flow/BDR/=.
- Add archive metadata such as =ARCHIVED_FROM= and =ARCHIVE_REASON= on the destination record instead of encoding archive status as a hidden search/query side effect.
- Do not archive by deleting AST facts discovered through =asp org query= without writing the destination archive record first.

- If the query returns TODO headlines, treat them as =kind=task= and inspect the relevant selector before editing.
- If the query returns checkbox list items, treat them as =kind=checklistItem=; do not collapse them into task headlines.
- If the query returns a property drawer match, use the property key/value as the semantic anchor; do not invent SDD, Wendao, or agent-plan search modes.
- If the task needs a governing specification, choose it by purpose: =agent.sdd.v1= for software architecture and design drift, =agent.adr.v1= for a decision, =agent.bdd.v1= for behavior and acceptance, and =agent.prd.v1= for product requirements.
- If the task needs a new governing specification family, use =agent.spec-flow.v1= first. Define the theory, then add contract, template, example, skill registration, template index entry, validation, and adversarial review in that order.
- If no query result matches, narrow the AST query or inspect the linked contract/template before creating new structure.

** Plan Capture
Use =capture-plan= only for a non-mutating, reviewable Org entry plan. Capture kinds stay generic; plan contracts are caller-supplied Org properties and body content.

#+begin_src shell
asp org capture-plan \
  --kind task \
  --title <TITLE> \
  --target-file <ASP_ORG_ARTIFACTS_ROOT>/flow/plans/<PLAN_SLUG>.org \
  --outline Plans/Active \
  --tag plan \
  --property PLAN_ID=<PLAN_ID> \
  --body <OBJECTIVE>
#+end_src

The command renders an Org entry and application preconditions. It does not write the target file.
** Rules
- Preserve the semantic AST provider boundary: query Org tasks, checklist items, properties, planning timestamps, links, and source blocks as Org facts.
- Keep =ASP_ORG.org= as the single state skill entry; do not reintroduce a wrapper skill.
- Do not use =--kind agent-plan=; =agent-plan= is not a provider-owned capture kind.
- Do not reintroduce legacy guide labels such as =agent-plan-*=, =wendao-task-*=, =wendao-orgid=, =sdd-property=, or =plan-record=.
- Do not encode SDD, Wendao, or task archive behavior as hidden search/query side effects. Express them as AST queries first, then route mutations through an edit-plan or AST-patch contract.
- Do not place SDD, ADR, BDD, or PRD body content inside =agent.plan.v1= or =agent.execplan.v1=. Use =GOVERNING_CONTRACT= and =GOVERNING_REF= to link the plan to the specification record.
- Do not add a new governing specification family by editing the skill alone. First create or update a SpecFlow record, then add the contract, template, example, skill registration, template index entry, and validation evidence.
- Do not rewrite package-relative links as installation-root placeholders or repository-root paths.
